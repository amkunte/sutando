# Pointer Teacher

The capability that lets Sutando teach the user *anything they are doing on their own screen* by speaking and physically pointing at the relevant element — a port of Clicky's embodied "learn by doing" mechanism into Sutando. This `CONTEXT.md` scopes only the Pointer Teacher domain; it was created when that vocabulary was first resolved.

## Language

**Pointer Teacher**:
The whole capability — an embodied on-screen guide that watches the user's real screen, answers out loud, and flies a marker to the element it is talking about.
_Avoid_: assistant, copilot, "Clicky" (Clicky is the external app this is modelled on, not this feature)

**Point** (verb) / **Pointing**:
Moving the marker to a Target and drawing attention to it while narrating. A Point indicates; it never actuates the UI.
_Avoid_: click, highlight

**Target**:
The single on-screen element the user is being taught about, identified by intent ("the save button"), never supplied as a coordinate.
_Avoid_: element, widget, selector

**Pointer**:
The visible marker that travels to the Target (Clicky uses a flying triangle). It is the teacher's, not the user's, and never captures input.
_Avoid_: cursor (the OS cursor belongs to the user)

**Reactive teaching**:
The default mode — the user asks, the teacher answers and Points. No predetermined curriculum.
_Avoid_: freeform, chat

**Structured teaching**:
Optional mode where a pre-planned Lesson Spine walks the user stop-by-stop, interruptible by Reactive questions.
_Avoid_: tutorial, course (a generated browser course is a separate, existing capability)

**Lesson Spine**:
The ordered list of stops a Structured session follows; generated by analysing the subject (e.g. a codebase), not authored by hand.

## Relationships

- A **Pointer Teacher** session is either **Reactive** (default) or **Structured**; a **Structured** session is a **Lesson Spine** plus the ability to drop into **Reactive** at any stop.
- Every taught moment resolves one **Target**, then **Points** the **Pointer** at it while narrating.
- A **Target** is resolved intent → coordinate; the user never gives a coordinate.

## Example dialogue

> **Dev:** "When the user says 'how do I commit here', does the teacher click the button for them?"
> **Domain:** "No — it **Points**. It flies the **Pointer** to the commit button and says what to do. Pointing never actuates; the user stays in control. If it clicked, it would be an agent, not a teacher."
> **Dev:** "And 'teach me this codebase' — same thing?"
> **Domain:** "Same mechanism, **Structured** mode: a **Lesson Spine** over the repo, each stop a **Target** it **Points** at. The user can interrupt with a **Reactive** question and it resumes."

## Flagged ambiguities

- "cursor" meant both the OS pointer and the teacher's marker — resolved: the teacher's marker is the **Pointer**; it never captures input.
- "course" / "tutorial" overlapped with Sutando's existing `codebase-to-course` (browser HTML) — resolved: that is a separate capability; **Structured** mode reuses its *analysis* to build a **Lesson Spine** but teaches in the real app, not a web page.
- "Clicky" was used as the feature name — resolved: Clicky is the external precedent; the capability here is **Pointer Teacher**.
- "teach me the codebase in VSCode" was taken as the whole scope — resolved: that is one *example*; the capability is general ("learn anything by doing").
